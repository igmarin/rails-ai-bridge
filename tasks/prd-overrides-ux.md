# PRD: Overrides.md UX Improvements

## 1. Introduction / Overview

`config/rails_ai_bridge/overrides.md` is the primary mechanism for teams to inject
repo-specific knowledge (hot tables, auth scoping, mandatory spec types, compliance
rules) into the AI context files generated by rails-ai-bridge. Despite being one of
the most powerful features of the gem, it is consistently underused.

Three UX defects cause this underuse:

1. The file ships with an `<!-- rails-ai-bridge:omit-merge -->` guard on its first
   line. Teams must manually remove this line to activate overrides. Most teams
   forget, so their overrides are never merged.
2. The example content lives in a separate `overrides.md.example` file that most
   developers never read.
3. There is no feedback during `rails ai:bridge` or `rails ai:doctor` indicating
   whether overrides are active, empty, or silently skipped.

This feature addresses all three defects through changes to the install generator,
the doctor check system, and the `rails ai:bridge` CLI output. No new capability is
introduced — overrides already work correctly when activated.

---

## 2. Goals

- **G1.** Eliminate the manual "remove the guard line" activation step so overrides
  work out of the box as soon as a developer adds content.
- **G2.** Surface override status in `rails ai:doctor` so teams can see at a glance
  whether overrides are configured.
- **G3.** Print actionable, one-line feedback in the `rails ai:bridge` CLI output
  every time overrides are merged or found empty.
- **G4.** Consolidate the example content into the generated `overrides.md` stub
  itself so developers have immediate guidance without reading a second file.
- **G5.** Remove `overrides.md.example` as a separately generated file.

---

## 3. User Stories

**US-1 — Zero-friction activation**
As a developer setting up rails-ai-bridge, I want the overrides file to merge
automatically once I add content, so that I never have to remember to remove a
guard comment before my team's rules take effect.

**US-2 — Stub with embedded guidance**
As a developer opening `overrides.md` for the first time, I want to see commented
example sections inside the file itself, so that I know exactly what to write
without opening a separate example file.

**US-3 — Doctor feedback on overrides**
As a developer running `rails ai:doctor`, I want to see a dedicated override check
that tells me whether my overrides are active (and how many lines are merged) or
still empty, so that I can confirm the feature is working without manually checking
the generated files.

**US-4 — Bridge CLI merge feedback**
As a developer running `rails ai:bridge`, I want to see a single line confirming
which formats received merged overrides (and how many lines), or a nudge when the
file is empty, so that I know overrides are flowing through without opening each
generated file.

**US-5 — No more orphan example file**
As a developer looking at my project structure, I want a single `overrides.md` with
embedded examples rather than two files (`overrides.md` and `overrides.md.example`),
so that the feature is self-documenting.

## 4. Functional Requirements

### 4.1 Install Generator — new `overrides.md` stub

**FR-1.** The install generator must create `config/rails_ai_bridge/overrides.md`
without the `<!-- rails-ai-bridge:omit-merge -->` guard line.

**FR-2.** The generated stub must contain HTML comments (not rendered markdown)
that show example sections for: hot/large tables, authorization and scoping,
testing conventions, and performance/infrastructure. The comments must be clearly
labelled as examples and must not produce any merged content on their own.

**FR-3.** The generated stub must contain a brief plain-English header comment
(HTML comment) explaining that any non-comment content below it will be merged into
Copilot, Codex, and other supported AI context files.

**FR-4.** The install generator must NOT create `overrides.md.example` as a
separate file.

**FR-5.** If `overrides.md` already exists at the target path (e.g. upgrade
scenario), the generator must not overwrite it (existing idempotency behavior is
preserved).

### 4.2 `SharedAssistantGuidance` — guard detection update

**FR-6.** The `OMIT_MERGE_FIRST_LINE` constant and `mergeable_override_content?`
method must be retained for backwards compatibility (projects that already have the
guard in their `overrides.md` must continue to have it honored — their file will not
be merged until they remove the guard line manually).

**FR-7.** The existing `read_assistant_overrides` method must not change its return
semantics: it must still return `nil` when the file is absent, empty after stripping
HTML comments, or starts with the omit-merge guard.

**FR-8.** A new helper method `override_line_count` must return the number of
non-comment, non-blank lines in the overrides file (0 if the file is absent, empty,
or guarded). This method will be used by the doctor and CLI feedback.

### 4.3 `Doctor` — two new checks

**FR-9.** Add a new check named `"Overrides"` that:
- Returns `:pass` with message `"overrides.md is active — N lines merged"` when
  `override_line_count` > 0.
- Returns `:warn` with message `"overrides.md exists but contains only the stub
  template (no team rules added yet)"` and fix `"Edit
  config/rails_ai_bridge/overrides.md to add repo-specific rules"` when the file
  exists but `override_line_count` is 0 and the file does not have the omit-merge
  guard.
- Returns `:warn` with message `"overrides.md has the omit-merge guard — content
  will not be merged"` and fix `"Remove the first-line
  <!-- rails-ai-bridge:omit-merge --> comment to activate overrides"` when the file
  exists and starts with the omit-merge guard.
- Returns `:warn` with message `"config/rails_ai_bridge/overrides.md not found"` and
  fix `"Run rails ai:bridge:install or create the file manually"` when the file does
  not exist at all.

**FR-10.** The new `"Overrides"` check must be appended to the `CHECKS` constant in
`Doctor` so it appears in all `rails ai:doctor` runs.

### 4.4 `rails ai:bridge` CLI — merge feedback line

**FR-11.** After writing bridge files, the `rails ai:bridge` rake task (and all
per-format bridge tasks) must print a single override status line:
- When `override_line_count` > 0: print `[overrides] merged N lines into <formats>`
  where `<formats>` is a comma-separated list of the formats that received the
  merged content (e.g. `copilot, codex`). The line must use a consistent tag
  prefix `[overrides]` and must NOT use an emoji.
- When `override_line_count` is 0 and the file exists: print `[overrides]
  config/rails_ai_bridge/overrides.md is empty — add team conventions there`.
- When the file does not exist: print nothing (do not nudge on a fresh install
  before the file is created).

**FR-12.** The override status line must be printed once per `rails ai:bridge`
invocation regardless of how many formats were generated.

**FR-13.** Per-format tasks (e.g. `rails ai:bridge:copilot`) must also print the
override status line, scoped to their single format.

### 4.5 Reference count in `codex_support_serializer`

**FR-14.** The `.codex/README.md` rendered by `CodexSupportSerializer` must no
longer reference `overrides.md.example` or instruct users to remove the omit-merge
guard. It must instead describe the new default behavior: add content to
`overrides.md` and it will be merged automatically.

---

## 5. Non-Goals (Out of Scope)

- **No new merge targets.** Overrides are already merged into Copilot and Codex
  formats. This feature does not add new formats to the merge pipeline.
- **No time-based warnings.** The Cursor feedback mentioned "warn if omit-merge
  stays enabled beyond N days" — this is out of scope for this iteration.
- **No UI or dashboard.** Feedback is limited to the CLI (`rails ai:bridge` and
  `rails ai:doctor`).
- **No migration of existing `overrides.md` files.** Teams that already have the
  guard in their file will continue to need to remove it manually. The gem does not
  auto-modify existing user files.
- **No removal of the omit-merge detection logic.** `OMIT_MERGE_FIRST_LINE` and
  `mergeable_override_content?` must be retained for backward compatibility.
- **No changes to how overrides are merged into the generated files.** The merge
  logic in `repo_specific_guidance_section_lines` is not changed.
- **No changes to Gemini, Cursor rules, or Claude serializers** with respect to
  overrides (they do not currently consume overrides; that is out of scope).

## 6. Design Considerations

### New `overrides.md` stub template

The stub must be structured so that:
- All guide text is inside HTML comments and produces zero merged content.
- A developer can add real content immediately below the last comment block.
- The stub is short enough to read in one scroll (~40 lines max).

Rough structure:

```
<!--
  rails-ai-bridge: overrides.md
  Add team conventions below. Any non-comment content is automatically
  merged into Copilot (`.github/copilot-instructions.md`) and Codex (`AGENTS.md`).
  Run `rails ai:bridge` after editing.
-->

<!--
  EXAMPLES (delete or replace with real rules):

  ## Hot / large tables
  - Tables that must never be full-scanned in HTTP requests.
  - Required indexes for new filters or ORDER BY on high-volume tables.

  ## Authorization and scoping
  - Tenant/org scoping rules; PII fields; compliance constraints.

  ## Testing
  - Minimum spec types for certain changes.
  - Feature flags or contract tests if applicable.

  ## Performance / infrastructure
  - Read replicas, caching layers, known slow endpoints.
-->
```

### Doctor output style

New check appears alongside existing checks. Status icons follow the existing
convention (`:pass` = green check, `:warn` = yellow warning, `:fail` = red cross).

### CLI feedback style

Uses a `[overrides]` tag prefix consistent with a plain log style (no emoji, no
color codes required). Appears after `print_result` output and before the closing
"Done!" message.

## 7. Technical Considerations

- **`SharedAssistantGuidance` is a `module_function` module** — `override_line_count`
  must follow the same pattern (defined via `module_function`).
- **HTML comment stripping for line count** — `override_line_count` must count only
  lines that are outside HTML comment blocks (`<!-- ... -->`), after stripping blank
  lines. Multi-line HTML comments must be handled correctly.
- **Which formats receive overrides** — currently only `copilot` and `codex`
  serializers consume `repo_specific_guidance_section_lines`. The merge feedback
  line in the CLI must derive this list from the actual serializer behavior, not
  hard-code it.
- **Rake task context** — the override status line must be printed after
  `generate_context` returns so the line count reflects the file state at generation
  time. `SharedAssistantGuidance` is already available in the rake task context via
  `require "rails_ai_bridge"`.
- **Generator idempotency** — `create_file` in Rails generators is already
  idempotent when the target exists. No additional guard needed for FR-5.
- **`overrides.md.example` cleanup** — the generator method
  `create_assistant_overrides_template` must be updated; no separate migration is
  needed since the example file is not a gem asset.

## 9. Success Metrics

- **Zero manual activation required** — running the install generator and adding
  any non-comment content to `overrides.md` is sufficient to activate merging with
  no other steps.
- **Doctor coverage** — `rails ai:doctor` reports a `:pass` or `:warn` for
  `"Overrides"` on every run; no project is silent about override status.
- **CLI feedback present** — `rails ai:bridge` always prints exactly one
  `[overrides]` line per run (either "merged N lines" or the empty nudge).
- **Single file for overrides** — after install, only `config/rails_ai_bridge/overrides.md`
  exists; `overrides.md.example` is not created.
- **Backwards compatibility** — existing projects with the omit-merge guard in their
  `overrides.md` continue to have that guard respected (file is not merged).
- **Test coverage** — all new behavior is covered by unit tests; existing
  `shared_assistant_guidance_spec.rb` tests remain green.

## 10. Open Questions

- **OQ-1.** Should the `[overrides] config/rails_ai_bridge/overrides.md is empty`
  nudge also be shown during per-format tasks (`rails ai:bridge:copilot`,
  `rails ai:bridge:codex`) or only during the full `rails ai:bridge` run? Current
  leaning: show it in all bridge tasks that consume overrides.
- **OQ-2.** Should `override_line_count` be exposed as a public method on
  `SharedAssistantGuidance`, or should it be private / internal to `Doctor`?
  Current leaning: module_function on `SharedAssistantGuidance` so both Doctor and
  the rake task can use it without duplication.
- **OQ-3.** Should the example stub reference `rails ai:doctor` so developers know
  where to check override status? Current leaning: yes, one line in the header
  comment.
