```ruby
module RailsAiBridge
  class Service
    class << self
      def call(*args, **kwargs)
        new(*args, **kwargs).call
      end
    end

    def initialize(*args, **kwargs)
      # Standard initialization
    end

    def call
      raise NotImplementedError, "#{self.class} must implement #call"
    end
  end
end
```

**Result Object:**
```ruby
# lib/rails_ai_bridge/service/result.rb
module RailsAiBridge
  class Service::Result
    attr_reader :success, :data, :errors, :metadata

    def initialize(success, data: nil, errors: [], metadata: {})
      @success = success
      @data = data
      @errors = Array(errors)
      @metadata = metadata.freeze
    end

    def success? = success
    def failure? = !success
    def on_success(&block) = tap(&block) if success && block
    def on_failure(&block) = tap(&block) if failure? && block
    def to_h = {success: success, data: data, errors: errors, metadata: metadata}
  end
end
```

#### Service Design Patterns:

**AppIntrospectionService:**
```ruby
module RailsAiBridge
  module Services
    class AppIntrospectionService < RailsAiBridge::Service
      def self.call(app, only: nil, introspector_class: Introspector)
        new(app, introspector_class: introspector_class).call(only: only)
      end

      def initialize(app, introspector_class: Introspector)
        @app = app
        @introspector_class = introspector_class
      end

      def call(only: nil)
        introspector = @introspector_class.new(@app)
        introspection_result = introspector.call(only: only)
        
        return Service::Result.new(false, errors: ["Introspector must return a Hash"]) unless introspection_result.is_a?(Hash)
        return Service::Result.new(false, errors: ["Introspector returned error: #{introspection_result[:error]}"]) if introspection_result.key?(:error)
        
        Service::Result.new(true, data: introspection_result)
      rescue StandardError => e
        Service::Result.new(false, errors: [e.message])
      end
    end
  end
end
```

## Service Pattern Conventions ✅

This section documents the established conventions for service objects in Rails AI Bridge.

### 1. Service Class Structure

All services must follow this basic structure:

```ruby
module RailsAiBridge
  module Services
    class MyService < RailsAiBridge::Service
      # Class method for convenient calling
      def self.call(*args, **kwargs)
        new(*args, **kwargs).call
      end

      # Initialize with dependencies
      def initialize(dependency1, dependency2: DefaultClass)
        @dependency1 = dependency1
        @dependency2 = dependency2
      end

      # Main execution method
      def call
        # Business logic here
        Service::Result.new(true, data: result)
      rescue StandardError => e
        Service::Result.new(false, errors: [e.message])
      end
    end
  end
end
```

### 2. Result Pattern

All services must return `Service::Result` objects:

**Success Result:**
```ruby
Service::Result.new(
  true, 
  data: {models: ["User"], routes: {}},
  metadata: {timestamp: Time.now}
)
```

**Failure Result:**
```ruby
Service::Result.new(
  false,
  errors: ["Invalid input data"],
  metadata: {input: params}
)
```

### 3. Error Handling

**Standard Error Handling:**
```ruby
def call
  # Operation that might fail
  Service::Result.new(true, data: result)
rescue StandardError => e
  Service::Result.new(false, errors: [e.message])
end
```

**Specific Error Types:**
```ruby
rescue RailsAiBridge::ServiceErrors::ValidationError => e
  Service::Result.new(false, errors: [e.message])
end
```

### 4. Method Chaining

Use `on_success` and `on_failure` for chaining:

```ruby
result.on_success do |r|
  # Handle success
  r.data.transform_values(&:symbolize_keys)
end.on_failure do |r|
  # Handle failure
  log_error(r.errors.first)
end
```

### 5. Dependency Injection

**Constructor Injection:**
```ruby
def initialize(client, retry_count: 3)
  @client = client
  @retry_count = retry_count
end
```

**Default Dependencies:**
```ruby
def initialize(client: DefaultClient.new, retry_count: 3)
  @client = client
  @retry_count = retry_count
end
```

### 6. Testing Conventions

**Test Structure:**
```ruby
RSpec.describe RailsAiBridge::Services::MyService do
  describe ".call" do
    it "returns success result" do
      result = described_class.call(valid_input)
      expect(result.success?).to be(true)
    end

    it "returns failure result for invalid input" do
      result = described_class.call(invalid_input)
      expect(result.failure?).to be(true)
    end
  end
end
```

**Test Coverage Requirements:**
- 100% coverage for service methods
- Test both success and failure paths
- Test error handling
- Test edge cases

### 7. Documentation Standards

**Class Documentation:**
```ruby
# Service for performing X operations.
#
# Detailed description of what the service does.
#
# @example Basic usage
#   result = MyService.call(input)
#   if result.success?
#     puts result.data
#   end
class MyService < RailsAiBridge::Service
  # ...
end
```

**Method Documentation:**
```ruby
# Perform the main operation.
#
# @param arg1 [Type] description
# @param kwarg1 [Type] description (with default)
# @return [Service::Result] result object
# @raise [ServiceErrors::ValidationError] when input is invalid
def call(arg1, kwarg1: nil)
  # ...
end
```

### 8. Naming Conventions

**Service Classes:**
- Use `Service` suffix: `UserCreationService`
- Be specific about the operation: `PaymentProcessingService`
- Avoid generic names like `UserService`

**Method Names:**
- Main method: always `call`
- Private methods: use verb phrases like `validate_input`, `process_data`

**Variable Names:**
- `result` for Service::Result objects
- `data` for result data
- `errors` for error messages
- Use `@` prefix for instance variables

### 9. Performance Considerations

**Avoid N+1 Queries:**
```ruby
# Bad
users.each { |u| u.posts.load }

# Good
users.includes(:posts).each { |u| u.posts }
```

**Lazy Loading:**
```ruby
# Good - only load when needed
def call
  @data ||= expensive_operation
end
```

**Caching:**
```ruby
# Cache expensive operations
Rails.cache.fetch("expensive_key", expires_in: 1.hour) do
  expensive_operation
end
```

### 10. Logging and Monitoring

**Logging:**
```ruby
def call
  Rails.logger.info("Starting operation with params: #{params}")
  # ...
  Rails.logger.info("Operation completed in #{time_taken}ms")
rescue => e
  Rails.logger.error("Operation failed: #{e.message}")
  raise
end
```

**Metrics:**
```ruby
def call
  start_time = Time.now
  result = operation
  duration = Time.now - start_time
  track_metric("service.duration", duration)
  result
end
```

## Migration Guide

### From Procedural to Service Pattern

**Before:**
```ruby
def process_user(user)
  if user.valid?
    {success: true, data: user}
  else
    {success: false, errors: user.errors}
  end
end
```

**After:**
```ruby
class UserProcessingService < RailsAiBridge::Service
  def initialize(user)
    @user = user
  end

  def call
    if @user.valid?
      Service::Result.new(true, data: @user)
    else
      Service::Result.new(false, errors: @user.errors)
    end
  end
end

# Usage
result = UserProcessingService.call(user)
```

### Error Handling Migration

**Before:**
```ruby
begin
  process_data
rescue => e
  {error: e.message}
end
```

**After:**
```ruby
result = DataProcessingService.call(data)
if result.failure?
  handle_error(result.errors.first)
end
```

## Best Practices

### 1. Single Responsibility Principle
Each service should do one thing well.

### 2. Fail Fast
Validate inputs early and fail with clear error messages.

### 3. Immutable Results
Never modify result data after creation.

### 4. Consistent Interfaces
All services should have the same calling convention.

### 5. Test in Isolation
Mock dependencies to test service logic independently.

### 6. Document Assumptions
Clearly document preconditions and postconditions.

### 7. Handle Edge Cases
Consider empty inputs, nil values, and boundary conditions.

### 8. Performance Matters
Optimize critical paths and cache expensive operations.

### 9. Monitor and Log
Instrument services for observability.

### 10. Keep It Simple
Avoid unnecessary complexity in service implementations.

## Common Anti-Patterns

### ❌ Mixing Concerns
```ruby
# Bad - does too much
class UserService
  def create_user; end
  def update_user; end
  def delete_user; end
  def send_email; end
end
```

### ❌ Inconsistent Returns
```ruby
# Bad - mixed return types
def call
  if success
    data
  else
    nil
  end
end
```

### ❌ Silent Failures
```ruby
# Bad - swallows errors
def call
  operation
rescue
  nil # Don't do this!
end
```

### ❌ Global State
```ruby
# Bad - uses global state
def call
  $global_var = something
end
```

### ❌ Complex Nesting
```ruby
# Bad - deeply nested logic
def call
  if condition1
    if condition2
      if condition3
        # ...
      end
    end
  end
end
```

## Decision Log

### Why Service Objects?
- **Testability**: Easy to test in isolation
- **Reusability**: Services can be reused across different contexts
- **Maintainability**: Clear separation of concerns
- **Consistency**: Standardized error handling and results

### Why Not Use Plain Methods?
- No standardized error handling
- Harder to test and mock
- Inconsistent return types
- No clear pattern for success/failure

### Why Not Use ActiveInteraction?
- Adds external dependency
- Less control over error handling
- Different result pattern
- More complex for our use case

### Why Not Use Dry::Monads?
- Functional programming style doesn't fit our codebase
- Steeper learning curve
- Overkill for our needs
